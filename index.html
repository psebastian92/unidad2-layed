Programas Robustos: Validación de Datos

En la Unidad 1 aprendiste a escribir programas que funcionan… cuando el usuario se porta bien. En esta unidad aprendemos a escribir programas que sobreviven al mundo real: usuarios que dejan campos vacíos, que ingresan edades negativas, que eligen la opción 7 en un menú de 4.

¿Qué es un programa robusto? Un programa que no confía en los datos que recibe: los revisa antes de usarlos, y si están mal, avisa y vuelve a pedirlos. Todos los sistemas serios funcionan así: probá dejar tu usuario vacío en Instagram, o ponerle fecha de nacimiento año 3050 a un formulario. No te dejan. Eso es validación.
🧠 La nueva pregunta que se suma a las de siempre Ya conocés las tres de la U1: ¿qué datos necesito? ¿qué quiero mostrar? ¿qué pasos hay en el medio? Esta unidad agrega la cuarta, la del programador desconfiado: ¿qué puede venir mal en cada dato?

Herramientas nuevas de esta unidad: cero. Todo se construye con lo que ya sabés de la U1: while, do-while, operadores lógicos y Scanner. Lo nuevo es cómo pensás el programa.

Tema 01

El programa ingenuo

Este programa es 100% correcto según la Unidad 1: compila, ejecuta, muestra los datos. Copialo en Eclipse y probalo… pero probalo como usuario malicioso.

import java.util.Scanner;

public class RegistroIngenuo {
    public static void main(String[] args) {
        Scanner entrada = new Scanner(System.in);

        System.out.println("Ingrese su nombre: ");
        String nombre = entrada.nextLine();

        System.out.println("Ingrese su edad: ");
        int edad = entrada.nextInt();

        System.out.println("\n--- Datos registrados correctamente ---");
        System.out.println("Nombre: " + nombre);
        System.out.println("Edad: " + edad);
    }
}
🔨 Desafío: rompelo Ejecutalo tres veces y probá:
① Apretá Enter sin escribir nada en el nombre.
② Ingresá edad -5.
③ Ingresá edad 900.
¿Qué hace el programa en cada caso?

Spoiler de lo que vas a ver:

Ingrese su nombre: Ingrese su edad: -5 --- Datos registrados correctamente --- Nombre: Edad: -5

El programa registró "correctamente" a una persona sin nombre y con edad negativa. No tiró error, no se quejó, no explotó: hizo algo peor. Aceptó basura y la trató como si fuera un dato válido. Imaginate esto en el sistema de un hospital, de un banco o del SIU de una facultad.

🧠 El problema no es el usuario El usuario puede equivocarse, apurarse o directamente querer romper el sistema. Eso no lo controlás. Lo que sí controlás es tu programa: la responsabilidad de rechazar datos inválidos es siempre del código, nunca del usuario. Grabate esa frase, porque en Java Web va a valer oro.
Tema 02

El patrón de validación

La solución tiene forma de receta, y la vas a usar mil veces: pedir el dato, y mientras esté mal, avisar y volver a pedirlo. ¿Qué estructura de la U1 repite algo mientras se cumple una condición? Exacto: el while.

El esqueleto en Java

// 1. PEDIR el dato (una primera vez, ANTES del while)
dato = entrada.next...();

// 2. MIENTRAS el dato esté MAL...
while (condición que describe el dato INVÁLIDO) {
    // 3. AVISAR qué está mal (con System.err, en rojo)
    System.err.println("Error: ...");
    // 4. VOLVER a pedir
    dato = entrada.next...();
}
// Acá abajo, el dato es válido SÍ o SÍ
⚠️ El detalle que confunde a todos: la condición se escribe AL REVÉS La condición del while describe el dato inválido, no el válido. Si la edad válida es de 0 a 130, el while dice edad < 0 || edad > 130 — o sea, "mientras esté fuera del rango, seguí pidiendo". Es el mismo razonamiento del for decreciente de la U1: pensá qué condición mantiene vivo el bucle.
🧠 ¿Por qué se pide el dato DOS veces en el código? Una vez antes del while (si no, la variable estaría vacía y el while no tendría qué analizar) y otra adentro (si no, el bucle repetiría el cartel de error infinitamente sin dejar corregir). Sacá mentalmente cada una y pensá qué pasaría. Esto cae en el examen escrito, seguro. 😏
Tema 03

Validar textos

El error más común con un texto: que venga vacío (el usuario apretó Enter de una) o lleno de espacios. Java tiene un detector exacto para eso: isBlank().

// --- Bloque para el nombre ---
System.out.println("Ingrese su nombre: ");
String nombre = entrada.nextLine();

// Validación: mientras el nombre esté vacío o solo tenga espacios, repite la petición
while (nombre.isBlank()) {
    System.err.println("Error: El nombre no puede quedar vacío. Inténtelo de nuevo:");
    nombre = entrada.nextLine();
}
Ingrese su nombre: Error: El nombre no puede quedar vacío. Inténtelo de nuevo: Error: El nombre no puede quedar vacío. Inténtelo de nuevo: Zoe

Fijate el patrón completo en acción: pedir → while con la condición de lo inválido → avisar en rojo → volver a pedir. El usuario puede insistir con Enter veinte veces: el programa no avanza hasta tener un nombre de verdad.

⚠️ Acá aparece nextLine(), que no es next() En la U1 usábamos next(), que lee una palabra y nunca devuelve vacío (se queda esperando). Para validar vacíos necesitamos nextLine(), que lee la línea entera, espacios incluidos — y por eso puede venir en blanco. Ojo con mezclarlos: si usás nextLine() después de un nextInt(), se traga el Enter que quedó flotando y parece que "saltea" el ingreso. Regla práctica de esta unidad: pedí los textos primero, los números después — como en el ejemplo completo del Tema 04.

Validar opciones de texto: equalsIgnoreCase()

Otro caso típico: una respuesta que solo puede ser "si" o "no". Para comparar Strings nunca se usa == (eso compara otra cosa que vas a entender en POO); se usa equals(), o mejor, equalsIgnoreCase(), que ignora mayúsculas y minúsculas:

System.out.println("¿Confirma la inscripción? (si/no)");
String respuesta = entrada.nextLine();

// Mientras NO sea "si" NI "no" (en cualquier combinación de mayúsculas)...
while (!respuesta.equalsIgnoreCase("si") && !respuesta.equalsIgnoreCase("no")) {
    System.err.println("Error: responda 'si' o 'no'.");
    respuesta = entrada.nextLine();
}

if (respuesta.equalsIgnoreCase("si")) {
    System.out.println("Inscripción confirmada.");
} else {
    System.out.println("Inscripción cancelada.");
}
🧠 Leé la condición en voz alta !respuesta.equalsIgnoreCase("si") && !respuesta.equalsIgnoreCase("no") se lee: "mientras la respuesta NO sea 'si' Y TAMPOCO sea 'no'". Los dos ! y el && trabajando juntos: repaso directo del Tema 06 de la U1.
Tema 04

Validar rangos numéricos

Con los números, el enemigo no es el vacío: son los valores absurdos. Edades negativas, notas de 47 puntos, precios de -100 pesos. La herramienta: el while con operadores de comparación.

El programa completo: registro con validación

Este es el programa ingenuo del Tema 01, pero ya curado. Compará los dos, línea por línea:

import java.util.Scanner;

public class Validaciones {
    public static void main(String[] args) {
        Scanner entrada = new Scanner(System.in);

        /* SISTEMA DE INGRESO DE EDADES CON VALIDACIÓN */

        // --- Bloque para el nombre ---
        System.out.println("Ingrese su nombre: ");
        String nombre = entrada.nextLine();

        while (nombre.isBlank()) {
            System.err.println("Error: El nombre no puede quedar vacío. Inténtelo de nuevo:");
            nombre = entrada.nextLine();
        }

        // --- Bloque para la edad ---
        System.out.println("Ingrese su edad: ");
        int edad = entrada.nextInt();

        // Validación de rango: la edad debe ser coherente (entre 0 y 130 años)
        while (edad < 0 || edad > 130) {
            System.err.println("Error: Edad no válida. Ingrese un valor entre 0 y 130:");
            edad = entrada.nextInt();
        }

        // --- Salida de datos finales ---
        System.out.println("\n--- Datos registrados correctamente ---");
        System.out.println("Nombre: " + nombre);
        System.out.println("Edad: " + edad);
    }
}
Ingrese su nombre: Thiago Ingrese su edad: -5 Error: Edad no válida. Ingrese un valor entre 0 y 130: 900 Error: Edad no válida. Ingrese un valor entre 0 y 130: 17 --- Datos registrados correctamente --- Nombre: Thiago Edad: 17

Ahora sí: el "correctamente" del mensaje final es verdad. Ningún dato llegó ahí sin pasar la aduana.

La receta para escribir la condición del rango

Pensá primero lo válido, después invertilo para el while:

DatoLo válidoLa condición del while (lo inválido)
Edadde 0 a 130edad < 0 || edad > 130
Notade 1 a 10nota < 1 || nota > 10
Opción de menúde 1 a 4opcion < 1 || opcion > 4
Preciomayor a 0precio <= 0
Día del mesde 1 a 30dia < 1 || dia > 30
⚠️ El error clásico: usar && donde va || edad < 0 && edad > 130 pregunta si la edad es menor que 0 y al mismo tiempo mayor que 130. Ningún número puede ser las dos cosas a la vez, así que esa condición es siempre falsa y el while nunca valida nada. Un dato está fuera del rango si se escapa por abajo O por arriba: siempre ||.

Tema 05

Menús a prueba de balas

El menú de opciones es EL lugar donde la validación se vuelve indispensable, porque un menú roto rompe todo el programa. Y acá el do-while de la U1 encuentra su trabajo soñado: mostrar el menú al menos una vez, y repetirlo mientras la opción sea inválida.

int opcion;

do {
    System.out.println("\n===== KIOSCO DEL COLE =====");
    System.out.println("1. Ver productos");
    System.out.println("2. Comprar");
    System.out.println("3. Ver mi saldo");
    System.out.println("4. Salir");
    System.out.println("Elija una opcion: ");

    opcion = entrada.nextInt();

    if (opcion < 1 || opcion > 4) {
        System.err.println("Error: la opcion " + opcion + " no existe.");
    }

} while (opcion < 1 || opcion > 4);

// Llegado acá, la opción es válida: la procesamos con switch (¡U1, Tema 08!)
switch (opcion) {
    case 1: System.out.println("Alfajores, gaseosas, chicles..."); break;
    case 2: System.out.println("Elegiste comprar."); break;
    case 3: System.out.println("Tu saldo es de $2500."); break;
    case 4: System.out.println("Hasta la proxima!"); break;
}
===== KIOSCO DEL COLE ===== 1. Ver productos 2. Comprar 3. Ver mi saldo 4. Salir Elija una opcion: 8 Error: la opcion 8 no existe. ===== KIOSCO DEL COLE ===== 1. Ver productos 2. Comprar 3. Ver mi saldo 4. Salir Elija una opcion: 3 Tu saldo es de $2500.
🧠 ¿Por qué do-while acá y while en los otros temas? Porque el menú tiene que mostrarse al menos una vez, sí o sí — es exactamente la especialidad del do-while que viste en la U1. En cambio, para validar un dato ya pedido, el while común alcanza. Elegir el bucle correcto para cada situación: eso también es diseñar el programa antes de tipearlo.
💡 Mirá todo lo que se juntó acá En este ejemplo trabajan juntos: do-while (U1-T09), operadores lógicos (U1-T06), Scanner (U1-T05), switch con break (U1-T08) y el patrón de validación (U2-T02). Los programas de verdad son esto: las piezas de siempre, bien combinadas. Este menú validado es el corazón de los sistemas que vas a armar en la Unidad 4 con listas.
Tema 06

La caja que no podemos cerrar (todavía)

Seamos honestos: nuestro registro validado del Tema 04 sigue teniendo un punto débil. Probá esto: cuando te pida la edad, escribí hola.

Ingrese su nombre: Milagros Ingrese su edad: hola Exception in thread "main" java.util.InputMismatchException at java.base/java.util.Scanner.throwFor(Scanner.java:964) at java.base/java.util.Scanner.next(Scanner.java:1619) at java.base/java.util.Scanner.nextInt(Scanner.java:2284) at validaciones.Validaciones.main(Validaciones.java:21)

💥 El programa explotó. Esa pantalla roja se llama excepción (acá, una InputMismatchException: "lo que ingresaste no coincide con lo que esperaba"). Y fijate el detalle cruel: explotó antes de llegar a nuestro while. Nuestra validación de rango nunca tuvo chance de actuar, porque el error ocurre en el momento mismo en que nextInt() intenta convertir "hola" en número.

⚠️ ¿Por qué el while no puede resolver esto? Porque el while compara valores: necesita que la edad ya sea un número para preguntarle si está entre 0 y 130. Pero "hola" no puede convertirse en número — el problema no es el valor del dato, es su tipo. Es una categoría de error completamente distinta, y las herramientas de esta unidad no la alcanzan.

El spoiler: try-catch

Java tiene una herramienta específica para los errores que no se pueden prevenir comparando valores: se llama try-catch, y funciona como una red de seguridad — "intentá (try) esta operación riesgosa, y si explota, atrapala (catch) y decidí qué hacer, en lugar de morir con la pantalla roja".

La vamos a estudiar más adelante, justo antes de Java Web. ¿Por qué ahí? Porque en la web esto deja de ser un caso raro y pasa a ser el pan de cada día: todo lo que llega de una página web llega como texto y hay que convertirlo; la base de datos puede estar caída justo cuando tu programa la necesita; una ruta mal escrita hace fallar la conexión entre páginas. Son errores que no dependen de tu código sino del mundo exterior — y contra el mundo exterior, el if no alcanza.
